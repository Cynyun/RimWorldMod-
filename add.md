# Mod 管理器 —— 新增功能需求说明（增量开发专用）

> 项目技术栈：WPF + .NET（Visual Studio 开发）  
> UI 框架：**WPF-UI**（https://github.com/lepoco/wpfui）  
> ⚠️ **重要原则**：本次仅新增指定功能，**不得修改任何与以下需求无关的现有代码**，包括但不限于：  
> - 现有业务逻辑（非本需求涉及部分）  
> - 配置加载、日志系统、异常处理等基础设施  
> - 未提及的 UI 页面或控件行为  

---

## 🎯 新增功能范围（仅限以下内容）

### 1. **UI 布局调整（最小必要改动）**
- 在主界面 Mod 列表区域，**替换为两个可切换的视图**：
  - 视图 A：显示游戏 `mods/` 目录下的本地 Mod
  - 视图 B：显示 `steamcmd/steamapps/workshop/content/294100/` 下的 Workshop Mod
- 使用 **WPF-UI 提供的控件**（如 `ToggleSwitch`、`Button` 或 `NavigationView`）实现视图切换
- **每个列表项必须包含复选框**，支持多选
- 其他 UI 元素（如标题、操作按钮区）可保留原布局，仅做必要适配

> ✅ 允许：调整 XAML 布局结构以容纳双列表切换  
> ❌ 禁止：重写整个主窗口、更换主题、修改未涉及的控件样式

---

### 2. **Mod 名称解析与展示**
- 对于 Workshop Mod（即 `294100/{ID}` 文件夹）：
  - 若存在 `modinfo.json`（或类似标准元数据文件），从中读取 Mod 名称用于显示
  - 若无有效名称，显示为“未知 Mod (ID: {ID})”
- **本地 Mod 列表**需尝试关联其是否来源于 Workshop，并在名称旁标注来源 ID（如有）

> ✅ 允许：新增辅助方法用于读取 `modinfo.json`  
> ❌ 禁止：修改现有 Mod 加载逻辑中与路径扫描无关的部分

---

### 3. **新增“导入”操作**
- 在 Workshop 视图下，提供 **“导入选中 Mod 到游戏”** 操作按钮（使用 WPF-UI `Button`）
- 行为要求：
  - 将选中的 Workshop Mod **完整复制**到游戏 `mods/` 目录
  - 目标文件夹名为解析出的 Mod 显示名称（若冲突则自动追加序号）
  - **不得修改或删除** SteamCMD 原始文件
- 操作完成后，应刷新本地 Mod 列表（仅触发已有刷新机制，不重写）

> ✅ 允许：调用现有文件复制/目录创建工具类（如有）  
> ❌ 禁止：引入新的 I/O 抽象层或重构文件操作逻辑

---

### 4. **新增本地持久化映射记录**
- 引入一个**轻量级本地存储机制**（如 JSON 文件），用于记录：
  - Workshop ID ↔ Mod 显示名称
  - 是否已导入到本地
  - 实际使用的本地文件夹名
- 此数据用于：
  - Workshop 列表显示名称
  - 本地列表反查来源 ID
  - 避免重复导入
- 存储位置必须为用户配置目录（如 `%AppData%\{AppName}\`）

> ✅ 允许：新增一个独立的数据访问类（如 `ModRegistryService`）  
> ❌ 禁止：修改现有配置管理器、数据库连接、序列化框架等

---

### 5. **交互反馈**
- 所有用户操作（导入成功/失败、名称冲突等）必须通过 **WPF-UI 的 `Snackbar` 或 `Dialog`** 组件反馈
- 不得使用 `MessageBox.Show()` 或其他原生弹窗

---

## 📦 发布与工程约束

- 所有新增 UI 必须使用 **WPF-UI 控件**，尽可能不使用 `System.Windows.Controls` 原生控件
- 新增代码应尽量**解耦**，避免侵入现有模块
- 项目仍通过 **Visual Studio 发布**（ClickOnce / MSIX），确保新增文件（如缓存 JSON）不会被误打包进安装目录只读区
- 新增功能应可独立测试，不影响原有功能流程
